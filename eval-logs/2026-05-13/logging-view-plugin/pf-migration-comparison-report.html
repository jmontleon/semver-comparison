🎨 CSS Token / Variable Migration pf-codemods — CRITICAL ISSUE in fix-engine

PF6 change: PF6 introduced a new design token system. Global tokens use the --pf-t--global-- prefix. PF6 provides both semantic aliases (e.g., --pf-t--global--spacer--sm) and numeric tokens (e.g., --pf-t--global--spacer--200). The semantic aliases are defined in the PF6 stylesheet as CSS custom properties that resolve to the numeric tokens.

🔴 Critical Finding: fix-engine introduces fabricated CSS tokens

The fix-engine branch attempts to "update" CSS tokens to what it believes are PF6 numeric equivalents, but invents token names that do not exist in the PF6 stylesheet. These tokens were verified against @patternfly/patternfly@6.4.0's patternfly-base-no-globals.css:

Token Used by fix-engineExists in PF6?What main uses (valid)
--pf-t--global--FontSize--sm❌ NO--pf-t--global--font--size--sm ✅
--pf-t--global--FontSize--xs❌ NO--pf-t--global--font--size--xs ✅
--pf-t--global--FontSize--md❌ NO--pf-t--global--font--size--md ✅
--pf-t--global--BackgroundColor--100❌ NO--pf-t--global--background--color--secondary--default ✅
--pf-t--global--Background-color--100❌ NO--pf-t--global--background--color--primary--default ✅
--pf-t--global--BorderColor--100❌ NO--pf-t--global--border--color--on-secondary ✅
--pf-t--global--BorderWidth--sm❌ NO--pf-t--global--border--width--regular ✅

Impact: All 10+ CSS rules using these fabricated tokens will silently resolve to the CSS initial value (typically 0, transparent, or normal) at runtime. This means:

  • Font sizes will collapse to browser defaults on affected elements
  • Background colors will disappear
  • Border widths/colors will disappear
  • Spacing will collapse

Main's tokens are valid PF6 aliases

The semantic token names that main uses (e.g., --pf-t--global--spacer--sm) are valid in PF6. They are defined as aliases:

/* From @patternfly/patternfly@6.4.0 patternfly-base-no-globals.css */
--pf-t--global--spacer--sm: var(--pf-t--global--spacer--200);
--pf-t--global--spacer--xs: var(--pf-t--global--spacer--100);
--pf-t--global--spacer--md: var(--pf-t--global--spacer--300);
--pf-t--global--font--size--sm: var(--pf-t--global--font--size--200);

Additional CSS bug in fix-engine

In precision-time-picker.css:

/* main (valid CSS): */
padding-top: var(--pf-t--global--spacer--xs);

/* fix-engine (INVALID CSS — quotes around the value): */
padding-top: 'var(--pf-t--global--spacer--100)';

The quotes make this an invalid CSS value. The browser will ignore the entire declaration.

Spacer token changes (valid but unnecessary)

fix-engine also converts semantic spacer names to numeric equivalents (e.g., --spacer--sm → --spacer--200). These conversions are functionally correct since both tokens exist, but are unnecessary since the semantic names are valid PF6 aliases. This churn adds no value.

Winner: pf-codemods — makes zero CSS changes, preserving the valid PF6 tokens already on main. fix-engine's CSS "migration" introduces critical visual bugs.
