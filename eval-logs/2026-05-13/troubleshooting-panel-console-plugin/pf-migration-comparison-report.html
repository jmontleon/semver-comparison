🔘 Button Component — icon Prop Migration

PatternFly 6 requires that icon-only buttons use the icon prop instead of passing icon elements as children. The PF6 codemods tool (@patternfly/pf-codemods) handles this transformation automatically.

  <TimesCircleIcon className="tp-plugin__popover-close" />
</Button>

After (pf-codemods — correct PF6):

<Button icon={<TimesCircleIcon className="tp-plugin__popover-close" />}

  • Korrel8rPanel.tsx — SyncIcon ✅
  • Popover.tsx — TimesCircleIcon ✅

fix-engine not migrated

🪟 Modal Component Migration

PF6 has two Modal APIs:

  • New composable Modal — imported from @patternfly/react-core with ModalHeader, ModalBody, ModalFooter as children
  • Deprecated Modal — the old PF5 API, available at @patternfly/react-core/deprecated

The main branch already uses the new composable Modal API from the main import path.

pf-codemods-051326-2300 regression

Moves Modal and ModalVariant to @patternfly/react-core/deprecated. This is a regression — the codemod incorrectly assumed the old Modal API was in use, but main already uses the new composable API.

This does not break functionality (the deprecated API still works) but is semantically wrong and will need to be reverted.

  from '@patternfly/react-core';

Winner: fix-engine — preserves the correct composable Modal API.

🎨 CSS Token / Variable Migration

What changed in PF6

PF6 uses --pf-t-- prefixed design tokens. Within PF6, both semantic names (--pf-t--global--spacer--sm) and numeric names (--pf-t--global--spacer--200) are valid. They map as follows:

SemanticNumericValue
--pf-t--global--spacer--xs--pf-t--global--spacer--1000.25rem
--pf-t--global--spacer--sm--pf-t--global--spacer--2000.5rem
--pf-t--global--spacer--md--pf-t--global--spacer--3001rem

The main branch already uses valid PF6 --pf-t-- tokens throughout. Zero pf-v5 references exist in source.

pf-codemods-051326-2300 no change needed

Makes no CSS changes. All existing tokens are already valid PF6. This is correct — no migration work was needed.

fix-engine partial, inconsistent

Partially migrates semantic → numeric token names in some CSS files, but inconsistently:

  • korrel8rpanel.css: --spacer--xs → --spacer--100, --spacer--sm → --spacer--200, --spacer--md → --spacer--300 (but only in 2 of 4 rules)
  • popover.css: --background--color--primary--default → --background--color--100, --spacer--sm → --spacer--200 (but only in 2 of 4 uses)
  • Leaves --pf-t--global--spacer--sm and --pf-t--global--spacer--md in the same files — inconsistent mixing

Also removes the korrel8rtopology.css badge styling rules entirely (brand color overrides for topology node badges). This is a behavioral change, not a token migration.

Both naming conventions are valid in PF6. The inconsistent mixing within the same file is a maintenance concern but not a bug.

Winner: Tie — both are valid. pf-codemods is correct to not change already-valid tokens. fix-engine's partial update is unnecessary and inconsistent.
