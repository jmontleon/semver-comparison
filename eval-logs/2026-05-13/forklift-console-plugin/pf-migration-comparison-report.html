🚨 Modal Component Migration (Critical) Winner: main

What Changed in PF6

PF6 promoted the "Next" (composable) Modal to the default export. The composable Modal uses ModalHeader, ModalBody, ModalFooter as children. The old Modal (with title, actions props) moved to @patternfly/react-core/deprecated.

Critical Finding: pf-codemods creates a semantic mismatch

The codebase already uses the new composable Modal API on main. The pf-codemods branch incorrectly moves Modal to the deprecated import path — but the code continues to use the new API pattern (ModalHeader, ModalBody, ModalFooter as children).

Before (main — correct):

import {
  Modal,        // ← New composable Modal from @patternfly/react-core
  ModalBody,
  ModalFooter,
  ModalHeader,
  ModalVariant,
} from '@patternfly/react-core';

// Usage: composable pattern (correct with new Modal)
<Modal variant={ModalVariant.small} isOpen={true} onClose={closeModal}>
  <ModalHeader title="Delete" />
  <ModalBody>...</ModalBody>
  <ModalFooter>...</ModalFooter>
</Modal>

After (pf-codemods — INCORRECT):

import {
  ModalBody,
  ModalFooter,
  ModalHeader,
} from '@patternfly/react-core';
import {
  Modal,        // ← OLD deprecated Modal (expects title/actions props)
  ModalVariant
} from '@patternfly/react-core/deprecated';

// Usage: STILL composable pattern (WRONG with deprecated Modal!)
<Modal variant={ModalVariant.small} isOpen={true} onClose={closeModal}>
  <ModalHeader title="Delete" />  // ← Rendered as body content, not header!
  <ModalBody>...</ModalBody>      // ← Nested inside deprecated Modal's own body!
  <ModalFooter>...</ModalFooter>  // ← Lost as body content!
</Modal>

⚠️ The deprecated Modal wraps ALL children in a ModalBoxBody and constructs its own header from the title prop. Passing ModalHeader, ModalBody, ModalFooter as children to the deprecated Modal results in all three being rendered as body content — producing a broken, headless, footerless modal at runtime.

🔴 eslint.config.ts Corruption (pf-codemods only) Critical Issue

Problem

The pf-codemods codemod tool corrupted eslint.config.ts in TWO places, replacing the string 'constructor' with the JavaScript internal string representation "function Object() { [native code] }".

Before (main — correct):

classes: ['field', 'constructor', 'private-instance-method', 'public-instance-method']

After (pf-codemods — CORRUPTED):

classes: ['field', "function Object() { [native code] }", 'private-instance-method', 'public-instance-method']

This breaks ESLint's @typescript-eslint/member-ordering rule configuration. The string "function Object() { [native code] }" is not a valid member group — ESLint will either error on this config or silently ignore the ordering rule.

Root cause: The codemod tool likely evaluated constructor as a JavaScript keyword/object property during AST transformation, substituting its runtime .toString() value instead of preserving the string literal.
