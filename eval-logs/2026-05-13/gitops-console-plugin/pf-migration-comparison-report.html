After (PF6 — codemods branch)

🎨 CSS Token / Design Variable Migration Mixed

What changed in PF6

PF6 introduces a completely new design token system. Old variables like --pf-v5-global--danger-color--100 are replaced with semantic tokens like --pf-t--global--color--status--danger--default. React token imports change from global_danger_color_100 to t_global_color_status_danger_default.

pf-codemods approach: placeholder tokens

- export const dangerColor = 'var(--pf-v5-global--danger-color--100)';
+ export const dangerColor = "var(--pf-t--temp--dev--tbd)"
  /* CODEMODS: original v5 color was --pf-v5-global--danger-color--100 */;

The codemod replaced 10 CSS variable references with --pf-t--temp--dev--tbd — a hot-pink placeholder designed to be visually obvious at runtime. The CODEMODS comments document the original values for manual follow-through. This is the expected codemod behavior per PatternFly docs, but the manual mapping step was not completed.

Bug in codemods output: The border property in ClusterGenerator.tsx and GenericGenerator.tsx lost its 1px solid prefix — the codemod replaced the entire string value, not just the CSS variable.

fix-engine approach: correct named imports

- import RedColor from '@patternfly/react-tokens/dist/esm/t_global_color_status_danger_100';
+ import { t_global_color_status_danger_default as RedColor } from '@patternfly/react-tokens';

Fix-engine updated 7 React token imports (in ResourceRow.tsx and icons.tsx) to use correct PF6 token names via named exports. These resolve to real CSS variables at runtime.

📥 Import Path Migration fix-engine wins

What changed in PF6

PF6 encourages named exports from package roots rather than deep path imports. Deep paths like @patternfly/react-tokens/dist/esm/t_global_* still work but are not the recommended pattern. Icon imports from dist/esm/icons/ are also discouraged.

pf-codemods branch (unchanged from main)

import YellowColor from '@patternfly/react-tokens/dist/esm/t_global_color_status_warning_100';

fix-engine uses the PF6-recommended named export pattern and also maps to the correct new token names (e.g., danger_100 → danger_default). This is the correct PF6 migration approach for token imports.

🎯 Scope Discipline & Minimality of Change pf-codemods wins decisively
