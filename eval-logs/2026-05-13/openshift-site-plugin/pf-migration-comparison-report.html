🔲 EmptyState Component Migration fix-engine wins

What changed in PF6

In PF6, EmptyStateHeader and EmptyStateIcon are no longer separate child components. Their functionality is absorbed into props on EmptyState itself:

  • EmptyStateHeader.titleText → EmptyState.titleText prop
  • EmptyStateHeader.headingLevel → EmptyState.headingLevel prop
  • EmptyStateIcon.icon → EmptyState.icon prop (pass component directly)

Before (PF5 — main branch)

// EmptyData/index.tsx
<EmptyState variant={EmptyStateVariant.lg} isFullHeight>
  {icon && <EmptyStateIcon icon={icon} />}
  <Title headingLevel="h2" size="lg">
    {message || t('Data not found')}
  </Title>
</EmptyState>

pf-codemods approach

// Still imports EmptyStateIcon!
import { ..., EmptyStateIcon, ... } from '@patternfly/react-core';

<EmptyState titleText={<Title headingLevel="h2" size="lg">
    {message || t('Data not found')}
  </Title>} variant={EmptyStateVariant.lg} isFullHeight>
  {icon && <EmptyStateIcon icon={icon} />}  ← still uses EmptyStateIcon as child!
</EmptyState>

fix-engine approach

// Removes EmptyStateIcon import
import { ..., EmptyStateVariant, Title } from '@patternfly/react-core';

<EmptyState variant={EmptyStateVariant.lg} isFullHeight icon={icon}>
  <Title headingLevel="h2" size="lg">
    {message || t('Data not found')}
  </Title>
</EmptyState>

🪟 Modal Migration fix-engine wins

What changed in PF6

PF6 introduces a new composable Modal API (promoted from @patternfly/react-core/next). The old Modal moves to @patternfly/react-core/deprecated. The recommended migration is:

  • Replace title prop with <ModalHeader title="..." />
  • Wrap content in <ModalBody>...</ModalBody>
  • Remove showClose, hasNoBodyWrapper (no longer valid on new Modal)
  • Import from @patternfly/react-core (not /deprecated)

Approach Comparison

Aspectpf-codemodsfix-engine
Import path @patternfly/react-core/deprecated (9 files) @patternfly/react-core (new API)
showClose prop Retained in all 9 files + CreateSitePage Removed from all files (1 remains on WarningModal — different component)
hasNoBodyWrapper prop Retained in 6 files Removed from all files
title prop → ModalHeader No — title prop still used Yes — ModalHeader used in CreateSitePage, Details.tsx
ModalBody wrapping No ModalBody used ModalBody added in ConnectorDetails, ListenerDetails, CreateSitePage, Details

fix-engine: Example (CreateSitePage.tsx)

import { Modal, ModalBody, ModalHeader, ModalVariant } from '@patternfly/react-core';

<Modal variant={ModalVariant.medium} isOpen={isOpen} aria-label="Form create site">
  <ModalHeader title={t('Create site')} />
  <ModalBody>
    <SiteForm onCancel={() => setIsOpen(false)} />
  </ModalBody>
</Modal>

pf-codemods: Example (CreateSitePage.tsx)

import { Modal, ModalVariant } from '@patternfly/react-core/deprecated';

<Modal title={t('Create site')} variant={ModalVariant.medium}
       isOpen={isOpen} showClose={false}>
  <SiteForm onCancel={() => setIsOpen(false)} />
</Modal>

Verdict: fix-engine adopts the new PF6 composable Modal API, which is the recommended path. pf-codemods merely moves the old Modal to /deprecated, retaining all old props (showClose, hasNoBodyWrapper, title). The deprecated approach is acceptable as a transitional step, but fix-engine's approach is more forward-looking and avoids deprecated API debt.

Note: fix-engine does not consistently add ModalBody to all modals (e.g., Connectors.tsx, Links.tsx, GetStarted.tsx still have content directly inside Modal without ModalBody). However, PF6 does not strictly require ModalBody for all use cases, and these modals build and render correctly.

📄 PageSection & PageNavigation Changes fix-engine wins

What changed in PF6

  • PageSectionVariants.light is removed — use "default" or "secondary"
  • PageNavigation is removed entirely
  • hasBodyWrapper is not a documented replacement for variant — the pf-codemods tool incorrectly substitutes it

Comparison

Changepf-codemodsfix-engine

Evidence: The pf-codemods branch uses hasBodyWrapper={false} as a replacement for variant={PageSectionVariants.light}. This is semantically wrong — hasBodyWrapper controls whether PageSection wraps its children in an additional body element, and is unrelated to the visual variant/background. fix-engine correctly maps light to default and selectively uses secondary where appropriate (GrantForm).
