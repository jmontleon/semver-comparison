🔲 Modal Component Migration

What changed in PF6

The old Modal component is deprecated in PF6 and moved to @patternfly/react-core/deprecated. The new Modal uses a composable pattern with ModalHeader, ModalBody, ModalFooter as children instead of title/footer props. The recommended migration path is:

  1. Move old Modal imports to /deprecated (interim — keeps code working)
  2. Refactor to new composable Modal API (full migration)

State on main

Main has 4 files that import Modal directly from @patternfly/react-core (wrong path in PF6):

  • NodeSelectorModal.tsx — uses old title prop API
  • DeleteInterfaceModal.tsx — uses new composable API (ModalBody/ModalFooter/ModalHeader as children)
  • ConfigurationDrawer.tsx — uses new composable API
  • NodesModal.tsx — uses new composable API

The other 4 modal-using files (DeleteModal, EditModal, PolicyEnactmentsDrawer, InterfaceDrawer) already use the new composable Modal API correctly on main but still import from the non-deprecated path.

pf-codemods-051326-2213 approach

- import { Modal, ModalVariant } from '@patternfly/react-core';
+ import { Modal, ModalVariant } from '@patternfly/react-core/deprecated';

Moves all 8 Modal imports to the /deprecated path. This is the correct interim step — it makes the code compile cleanly with PF6 and explicitly marks deprecated usage for future cleanup. No behavioral changes.

fix-engine approach

- <Modal title="Delete Interface" titleIconVariant="warning" footer={...}>
-   {body content}
- </Modal>
+ <Modal>
+   <ModalHeader title="Delete Interface" titleIconVariant="warning" />
+   <ModalBody>{body content}</ModalBody>
+   <ModalFooter>{footer content}</ModalFooter>
+ </Modal>

Restructures 6 files to use the new composable Modal API. This is the full migration target. However, fix-engine does not move the import to /deprecated — it keeps Modal imported from @patternfly/react-core. In PF6, the old Modal (with title/footer props) is in the deprecated path; the new composable Modal is the direct export. Since fix-engine rewrites to the new API, this works correctly.

📄 PageSection / PageBreadcrumb hasBodyWrapper

What changed in PF6

In PF6, PageSection and PageBreadcrumb now wrap content in a <div> wrapper by default (the "body wrapper"). To preserve PF5 behavior where no wrapper was added, you must set hasBodyWrapper={false}. This is flagged by the pageBreadcrumbAndSectionWarnUpdatedWrapperLogic codemod.

State on main

Only 1 of 10 PageSection/PageBreadcrumb instances has hasBodyWrapper={false} (DetailsPageTitle.tsx). The remaining 9 instances across 7 files are missing this prop.

+ <PageSection hasBodyWrapper={false}>
