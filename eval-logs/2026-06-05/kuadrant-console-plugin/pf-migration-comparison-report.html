🔲 Modal Component Migration Winner: goose

What changed in PF6

In PF5, there were two Modal APIs: the "old" Modal (with title prop, ModalBox/ModalBoxHeader sub-components) and the "next" Modal (composable Modal/ModalHeader/ModalBody/ModalFooter from @patternfly/react-core/next).

In PF6, the "next" version is promoted to the main export (@patternfly/react-core), and the old API is removed entirely. The /deprecated path does not export Modal in PF6.

Before (PF5 base)

// DropdownWithKebab.tsx — uses /next API
import { Modal, ModalBody, ModalFooter, ModalHeader } from '@patternfly/react-core/next';

// AddLimitModal.tsx, KuadrantAuthPolicyCreatePage.tsx — uses old API
import { Modal } from '@patternfly/react-core';
<Modal isOpen={isOpen} onClose={onClose} title={t('Add Limit')} width="50%">

// KuadrantAuthPolicyCreatePage.tsx — uses ModalBox sub-components
import { Modal, ModalBox, ModalBoxHeader, ModalBoxBody, ModalBoxFooter } from '@patternfly/react-core';

pf-codemods approach

// DropdownWithKebab.tsx — moved from /next to main with data-codemods markers
import {
	Modal /* data-codemods */,
	ModalBody /* data-codemods */,
	ModalFooter /* data-codemods */,
	ModalHeader /* data-codemods */,
} from '@patternfly/react-core';

// AddLimitModal.tsx, LimitSelect.tsx — routed to /deprecated (WRONG: Modal not in /deprecated)
import { Modal } from '@patternfly/react-core/deprecated';

// KuadrantAuthPolicyCreatePage.tsx — ModalBox etc. routed to /deprecated (also doesn't exist there)
import { Modal, ModalBox, ModalBoxHeader, ModalBoxBody, ModalBoxFooter } from '@patternfly/react-core/deprecated';

goose approach

// All files — use the new PF6 composable Modal API from main export
import { Modal, ModalHeader, ModalBody, ModalFooter } from '@patternfly/react-core';

// Old title-as-prop usage rewritten:
<Modal isOpen={isOpen} onClose={onClose}>
  <ModalHeader title={t('Add Limit')} />
  <ModalBody>...</ModalBody>
</Modal>

// ModalBox composites eliminated — flat structure:
<Modal isOpen={...} onClose={...} variant="medium">
  <ModalHeader title={title} />
  <ModalBody>...</ModalBody>
  <ModalFooter>...</ModalFooter>
</Modal>

📭 EmptyState Consolidation Winner: goose

What changed in PF6

In PF6, EmptyStateHeader and EmptyStateIcon are no longer exported separately. The titleText, headingLevel, and icon props are passed directly to EmptyState.

Before (PF5 base)

<EmptyState>
  <EmptyStateIcon icon={SearchIcon} />
  <Title headingLevel="h4" size="lg">No results found</Title>
  <EmptyStateBody>Try a different filter</EmptyStateBody>
</EmptyState>

pf-codemods approach

<EmptyState titleText={<Title headingLevel="h4" size="lg">No results found</Title>} icon={SearchIcon}>
  <EmptyStateBody>Try a different filter</EmptyStateBody>
</EmptyState>

⚠️ Passes a <Title> JSX element as titleText. While this works at runtime, the PF6 EmptyState type expects titleText to be a string, causing TS2322 errors in all 5 EmptyState instances.

goose approach

<EmptyState headingLevel="h4" titleText="No results found" icon={SearchIcon}>
  <EmptyStateBody>Try a different filter</EmptyStateBody>
</EmptyState>

✅ Passes titleText as a string and headingLevel as a separate prop, matching PF6 API exactly.

📄 PageSection variant Migration Winner: goose

What changed in PF6

In PF6, PageSection variant="light" is removed. The valid values are "default" and "secondary". The hasBodyWrapper prop is a separate concern about wrapper element rendering.

Comparison
