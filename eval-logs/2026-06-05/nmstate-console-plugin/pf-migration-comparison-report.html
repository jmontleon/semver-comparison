🔲 Modal Component Migration

What changed in PF6

The old Modal component was deprecated and moved to @patternfly/react-core/deprecated. The promoted (new) Modal from @patternfly/react-core uses a composable pattern with <ModalHeader>, <ModalBody>, and <ModalFooter> child components instead of title, header, footer, titleIconVariant props.

Source: PF6 Upgrade Guide — Modal section

6 Modals in codebase:

Modalgoose approachcodemods approach
NodeSelectorModal Promoted Modal + ModalHeader + ModalBody Modal from deprecated
DeleteInterfaceModal Promoted Modal + ModalHeader + ModalBody + ModalFooter Modal from deprecated
DeleteModal Promoted Modal + ModalHeader + ModalBody + ModalFooter Modal from deprecated
EditModal Promoted Modal + ModalHeader + ModalBody + ModalFooter Modal from deprecated
PolicyEnactmentsDrawer Promoted Modal + ModalHeader + ModalBody Modal from deprecated
InterfaceDrawer Promoted Modal + ModalHeader + ModalBody + ModalFooter Modal from deprecated (but kept header/footer props → TS error)

Code Example — DeleteModal

goose (promoted API):

import { Modal, ModalBody, ModalFooter, ModalHeader } from '@patternfly/react-core';

<Modal onClose={closeModal} variant="medium" isOpen={isOpen}>
  <ModalHeader title={t('Delete ...')} titleIconVariant="warning" />
  <ModalBody>...</ModalBody>
  <ModalFooter>...</ModalFooter>
</Modal>

codemods (deprecated path):

import { Modal } from '@patternfly/react-core/deprecated';

<Modal title={t('Delete ...')} titleIconVariant="warning">
  ...  // TS2322: titleIconVariant does not exist on deprecated Modal type
</Modal>

Verdict: The codemods branch moved Modal to /deprecated but did not restructure the component API — leaving titleIconVariant, header, and footer props that cause 5 TypeScript errors. The goose branch uses the promoted PF6 Modal correctly with composable children.

Winner: goose

📭 EmptyState API Migration

What changed in PF6

EmptyStateHeader and EmptyStateIcon were removed. Their functionality was absorbed into the EmptyState component via titleText, headingLevel, and icon props. EmptyStateActions blocks should be wrapped in <EmptyStateFooter>.

NNStateEmptyState comparison

goose — clean PF6:

<EmptyState variant="full" headingLevel="h5"
  titleText={t('No NodeNetworkStates found')} />

codemods — still uses Title:

<EmptyState titleText={<Title headingLevel="h5" size="lg">
    {t('No NodeNetworkStates found')}
  </Title>} variant={EmptyStateVariant.full}>
</EmptyState>

The codemods version passes a <Title> JSX element to titleText — this works but is not the intended PF6 pattern. The titleText prop is designed to accept a string; the heading level comes from the headingLevel prop on EmptyState itself.

PolicyListEmptyState comparison

goose correctly adds EmptyStateFooter wrapper around EmptyStateActions. Codemods does not add EmptyStateFooter — the actions remain as direct children of EmptyState, which may render differently in PF6.

Winner: goose — Cleaner and more correct PF6 EmptyState API usage.

🎯 Design Token Migration (color tokens)

What changed in PF6

Color tokens have no 1:1 mapping in PF6. The codemods tool replaces them with t_temp_dev_tbd placeholders that must be manually resolved to the correct PF6 token.

Source: PF6 Upgrade Guide — "Color tokens that don't have a defined 1:1 mapping will be replaced with a 'temp dev tbd' token"

Tokens in this codebase

Original PF5 tokengoose resolutioncodemods resolution
global_danger_color_200 t_global_color_status_danger_200 ✅ t_temp_dev_tbd ❌ (placeholder)
global_success_color_200 t_global_color_status_success_200 ✅ t_temp_dev_tbd ❌ (placeholder)

goose import:

import { t_global_color_status_danger_200 as dangerColor }
  from '@patternfly/react-tokens/dist/js/t_global_color_status_danger_200';

codemods import:

import { t_temp_dev_tbd as dangerColor
  /* CODEMODS: you should update this color token, original v5 token was global_danger_color_200 */
} from "@patternfly/react-tokens/dist/js/t_temp_dev_tbd";

The codemods branch leaves placeholder tokens that will render as hot pink (the PF6 dev TBD color) at runtime and cause the TS compiler to fail since the module path doesn't exist.

Winner: goose — Resolved to correct PF6 tokens.

📄 PageSection Variant Migration

What changed in PF6

PageSectionVariants.light was removed. The variant prop now only accepts "default" or "secondary".

Comparison

Approachgoosecodemods
Replacement for PageSectionVariants.light variant="default" hasBodyWrapper={false}

The goose approach (variant="default") is the documented PF6 replacement. The codemods approach (hasBodyWrapper={false}) is a different prop that controls whether PageSection renders a PageBody wrapper — it's not the semantic equivalent of the light variant. Both approaches compile, but goose's is more semantically correct.

Winner: goose — Semantically correct replacement.
