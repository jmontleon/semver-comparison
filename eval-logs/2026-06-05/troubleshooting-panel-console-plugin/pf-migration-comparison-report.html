📋 Modal Component Migration — Winner: semver/goose

What Changed in PF6

PF6 deprecates the old Modal API with title and actions props. The new composable Modal uses ModalHeader, ModalBody, and ModalFooter as children. The old API is available at @patternfly/react-core/deprecated but should be migrated for full PF6 compliance.

Before (PF5 base)

<Modal variant={ModalVariant.small}
  title={t('Custom time range')} isOpen onClose={onClose}
  actions={[
    <Button key="save" ...>{t('Save')}</Button>,
    <Button key="cancel" ...>{t('Cancel')}</Button>,
  ]}>
  {/* body content directly as children */}
</Modal>

pf-codemods

import { Modal, ModalVariant } from '@patternfly/react-core/deprecated';
// Modal usage unchanged — still uses title/actions props API

⚠️ Moved imports to deprecated path but did not actually restructure the Modal to the composable API. This is the minimum stopgap, not a real migration.

semver/goose

<Modal variant={ModalVariant.small} isOpen onClose={onClose}>
  <ModalHeader title={t('Custom time range')} />
  <ModalBody>
    {/* body content */}
  </ModalBody>
  <ModalFooter>
    <Button key="save" ...>{t('Save')}</Button>
    <Button key="cancel" ...>{t('Cancel')}</Button>
  </ModalFooter>
</Modal>

✅ Correctly restructures both TimeRangeModal and DurationModal to the PF6 composable pattern. Imports Modal, ModalBody, ModalFooter, ModalHeader from @patternfly/react-core (not deprecated).

This is a codemod execution bug where the tool resolved the JavaScript keyword constructor literally rather than treating it as a string value. This:

  • Will cause test reports to show garbled names
  • Represents an unintended functional change

The semver/goose branch does not touch types.spec.ts and preserves all test names correctly.

Context

web/cypress/views/utils.ts contains a getPFVersion() function that detects the running PatternFly version by probing CSS custom properties.

Base Branch Logic

if (style.getPropertyValue('--pf-v6-global--FontSize--md')) return 'v6';
else if (style.getPropertyValue('--pf-v5-global--FontSize--md')) return 'v5';

Both Branches Change

if (style.getPropertyValue('--pf-v6-global--FontSize--md')) return 'v6';
else if (style.getPropertyValue('--pf-t--global--font--size--md')) return 'v5';

Both branches make the same change: replacing the v5 fallback probe with a PF6 design token. However, the return value 'v5' is semantically incorrect when detecting --pf-t--* tokens (which are PF6). This appears to be a codemod artifact that neither branch cleaned up.
