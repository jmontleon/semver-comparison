📄 PageSection Migration Mixed — different strategies

What changed in PF6

In PF6, PageSection variant only accepts "default" or "secondary". The PageSectionVariants enum is still exported but only with these two values. The hasBodyWrapper prop (beta) controls whether children are wrapped in a PageBody. The type="nav" prop was removed.

pf-codemods approach

// Added hasBodyWrapper={false} to 22 PageSection instances
<PageSection hasBodyWrapper={false}
  variant={PageSectionVariants.default}>
  ...
</PageSection>

Keeps PageSectionVariants.default enum (still valid in PF6). Clean, consistent approach. hasBodyWrapper={false} ensures children aren't wrapped in an extra PageBody div, matching PF5 behavior.

semver/goose approach

Inconsistent — three different strategies used across files:

// Strategy 1: Replace PageSection with <section> (ApplicationDetailsTab)
<section className="co-m-pane__body">...</section>

// Strategy 2: Replace with <div> (ApplicationSourcesTab)
<div className="co-m-pane__body">...</div>

// Strategy 3: Wrap in <Page> + keep PageSection (ApplicationHistoryTab)
<Page>
  <PageSection variant={"default"}>...</PageSection>
</Page>

🎨 CSS Token Migration Winner: semver/goose

What changed in PF6

PF6 completely restructures CSS variables: --pf-v5-global--* → --pf-t--global--* (design tokens). The codemods tool replaces color tokens with a hot-pink placeholder --pf-t--temp--dev--tbd that must be manually resolved.

Before (main branch)

// In TypeScript
export const dangerColor = 'var(--pf-v5-global--danger-color--100)';
export const warningColor = 'var(--pf-v5-global--warning-color--100)';

// In SCSS
stroke: var(--pf-v5-global--success-color--100);
color: var(--pf-v5-global--Color--100) !important;

pf-codemods result

// Placeholder — NOT a real PF6 token
export const dangerColor = "var(--pf-t--temp--dev--tbd)"
  /* CODEMODS: original v5 color was --pf-v5-global--danger-color--100 */;

// Border value CORRUPTED
border: "var(--pf-t--temp--dev--tbd)"  // Missing "1px solid" prefix!
// Original: border: '1px solid var(--pf-v5-global--BorderColor--200)'

semver/goose result

// Resolved to real PF6 design tokens
export const dangerColor = 'var(--pf-t--global--color--status--danger--default)';
export const warningColor = 'var(--pf-t--global--color--status--warning--default)';

export const blueInfoColor = 'var(--pf-t--color--blue--40)';

// Border value PRESERVED correctly
border: '1px solid var(--pf-t--global--border--color--200)'

// SCSS also migrated

🪟 Modal Migration (Deprecated → New API) Winner: semver/goose

What changed in PF6

The old Modal was moved to @patternfly/react-core/deprecated. The new Modal (previously at /next) is now the primary export. It uses a composable pattern with ModalHeader, ModalBody, and ModalFooter instead of title/actions props.

pf-codemods

// NO CHANGE — still using deprecated imports
import { ModalVariant } from '@patternfly/react-core/deprecated';
import { Modal as PfModal } from '@patternfly/react-core/deprecated';

semver/goose

// Migrated to new composable Modal API
import { Modal as PfModal, ModalBody, ModalFooter, ModalHeader,
         ModalVariant } from '@patternfly/react-core';

<Modal isOpen={isModalOpen} onClose={handleModalClose} variant={ModalVariant.small}>
  <ModalHeader title={t('Add to favorites')} />
  <ModalBody>
    <Form id="confirm-favorite-form" onSubmit={handleConfirmStar}>...</Form>
  </ModalBody>
  <ModalFooter>
    <Button variant="primary">Save</Button>
    <Button variant="link">Cancel</Button>
  </ModalFooter>
</Modal>

Assessment: semver/goose correctly migrates to the new composable Modal API per PF6 guidance. pf-codemods leaves the deprecated import untouched.

⚛️ React 18 Type Compatibility Winner: semver/goose

What changed

React 18 removed the implicit children prop from React.FC and React.SFC (the latter was also removed entirely). Components that accept children must now explicitly declare children?: React.ReactNode in their props. React.ReactFragment was also removed in favor of React.ReactNode.

🚀 Merge Recommendation

Merge semver/goose/053026-2041 after fixing the following issues:

Required fixes before merge (estimated ~1-2 hours):

  1. Fix 19 build errors — Extend patches/apply-type-patches.sh to add children?: React.ReactNode to TableDataProps, CreateWithPermissionsProps, and StatusPopupSectionProps in the SDK types.
  2. Fix SplitItem component prop misuse in ResourceRow.tsx — revert to using children (the component prop sets the wrapper element type, not content).
  3. Review PageSection replacements — the 2 instances where PageSection was replaced with <div>/<section> (ApplicationDetailsTab, ApplicationSourcesTab) should use PageSection with hasBodyWrapper={false} instead to preserve PF styling.
  4. Review EmptyState titleText="" — empty string is technically valid but should contain a meaningful label for accessibility.

Optional improvements:

  • Validate --pf-t--color--blue--40 is a valid PF6 token (may need --pf-t--global--color--nonstatus--blue--300 instead)
  • Add .cache* to .gitignore
  • Migrate remaining 12 old-style pf-c-* SCSS selectors (shared tech debt)

Why not pf-codemods?

While pf-codemods builds cleanly, it represents only the first step of migration (running the codemod tool) without the essential second step (resolving placeholders, migrating CSS classes, updating deprecated APIs). The 12 --pf-t--temp--dev--tbd tokens would cause visual regressions at runtime (hot-pink or invisible colors). The 2 corrupted border values would break layout. It leaves 33+ pf-v5 references that will silently stop working. It's a starting point, not a shippable migration.
