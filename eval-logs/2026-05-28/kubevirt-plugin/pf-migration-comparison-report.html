💥 StackItem/SplitItem component= Crash Bug BLOCKER — semver/goose only

The Bug

The semver/goose branch moved StackItem and SplitItem children content into the component prop. This is fundamentally wrong.

How component actually works (PF6 source code)

// @patternfly/react-core/src/layouts/Stack/StackItem.tsx
export const StackItem = ({ component = 'div', children, ...props }) => {
  const Component = component as any;  // expects 'div', 'span', 'section'
  return <Component {...props}>{children}</Component>;
};

The component prop specifies the HTML element type to render (like 'div', 'span'). It does NOT accept JSX content.

What semver/goose does (WRONG)

// Before (correct — content as children):
<StackItem>{error.message}</StackItem>

// After (CRASH — string passed as element type):
<StackItem component={error.message} />
// React tries: <Taking snapshot of running VirtualMachine.>...</>  💥

// Before (correct — JSX as children):
<StackItem><AffinityDescriptionText /></StackItem>

// After (CRASH — React element passed as component constructor):
<StackItem component={<AffinityDescriptionText />} />
// React tries to call a React element as a function  💥

Impact

PatternCountRuntime Effect
StackItem component={<JSX />}46React error: element used as constructor
StackItem component={string}20Invalid HTML element names
SplitItem component={<JSX />}15React error: element used as constructor
SplitItem component={string/JSX}4Invalid HTML element names or crashes
Total: 85 crash sites across 40+ files
This is NOT a PatternFly 6 migration requirement. The component prop semantics did not change between PF5 and PF6. This bug was introduced by the AI agent's misunderstanding of the prop's purpose. Every page using Stack or Split layouts would crash at runtime.

Why does TypeScript not catch the StackItem bug?

Because the component prop is typed as React.ReactNode (not React.ElementType). This is a permissive type that accepts strings, JSX elements, and other React nodes. TypeScript sees no error, but React crashes at runtime when it tries to use a ReactNode as a constructor. This is a known PF type definition weakness.
