6. Detailed Analysis

6.1 Package Upgrades semver/goose

What changed in PF6

PatternFly 6 requires updating all @patternfly/* packages to v6-compatible versions. The react-charts package moved to v8.x with new peer dependencies (echarts, victory-* subpackages). New packages like @patternfly/react-drag-drop were introduced for drag-and-drop functionality previously bundled in core.

pf-codemods — No changes

package.json is identical to main. No version bumps, no new dependencies. This branch relies on the fact that main already has PF6 package versions.

semver/goose — Updated

@patternfly/react-charts: ~8.4.0 → ^8.5.1
@patternfly/react-drag-drop: ^6.0.0 (NEW)
@openshift-console/dynamic-plugin-sdk: 4.20.0 → ^4.21.0
victory-*: ^37.3.2 → ^37.3.6 (15 packages)
echarts: ^5.6.0 || ^6.0.0 (NEW)
victory-cursor-container: ^37.3.6 (NEW)
victory-zoom-container: ^37.3.6 (NEW)

⚠️ Issues:

  • react-helmet: ^6.1.0 → ^3.3.2 — Major version downgrade. Destructive bug.
  • @types/react: ^17.0.37 → ^18.3.0 — Type mismatch with React 17 runtime.
  • react/react-dom: Pinned 17.0.2 → Range ^17.0.1 — Pin loosened.

Verdict: semver/goose correctly adds missing PF6 peer dependencies (echarts, drag-drop, victory subpackages) that may be required for full PF6 chart functionality. However, the react-helmet downgrade and types/react mismatch are significant bugs. semver/goose still wins on net migration value despite the bugs.

6.2 Modal Deprecation pf-codemods

What changed in PF6

In PF6, the old Modal component (which used a single-component API) was moved to @patternfly/react-core/deprecated. The new composable Modal pattern (ModalHeader, ModalBody, ModalFooter) is the primary API in @patternfly/react-core.

Required change: import { Modal, ModalVariant } from '@patternfly/react-core' → import { Modal, ModalVariant } from '@patternfly/react-core/deprecated'

pf-codemods — ✅ Correctly migrated (5 files)

// Before (main):
import { Modal, ModalVariant, ModalBody, ... } from '@patternfly/react-core';

// After (pf-codemods):
import { ModalBody, ModalFooter, ModalHeader } from '@patternfly/react-core';
import { Modal, ModalVariant } from '@patternfly/react-core/deprecated';

Files: BaseModal.tsx, ConfirmModal.tsx, PodSelectorModal.tsx, CreateProjectModal.tsx, UserDefinedNetworkCreateModal.tsx

semver/goose — ❌ Not migrated

// Still on semver/goose:
import { Modal, ModalVariant, ModalBody, ... } from '@patternfly/react-core';

// Modal and ModalVariant NOT moved to /deprecated

All 5 Modal files remain unchanged — Modal and ModalVariant are still imported from the main @patternfly/react-core path.

Impact: While the build succeeds (PF6 re-exports deprecated components from main path with warnings), this is explicitly flagged in the PF6 upgrade guide and will generate console deprecation warnings at runtime. Official guidance says to use the /deprecated path as an intermediate step before fully migrating to the composable Modal API.

6.3 PageSection hasBodyWrapper pf-codemods

What changed in PF6

In PF6, <PageSection> and <PageBreadcrumb> now default to wrapping their content in a <div> body wrapper. To preserve PF5 behavior (no wrapper div), you must add hasBodyWrapper={false}.

pf-codemods — ✅ Comprehensive (24 instances, 21 files)

// Before:
<PageSection>
  {content}
</PageSection>

// After:
<PageSection hasBodyWrapper={false}>
  {content}
</PageSection>

Applied consistently to every <PageSection> and <PageBreadcrumb> in the codebase.

semver/goose — ❌ Not addressed

Zero hasBodyWrapper={false} additions. The branch instead removed <PageSection> entirely from 2 form files (NetworkAttachmentDefinitionForm.tsx, UserDefinedNetworkForm.tsx), which changes the layout rather than preserving it.

~22 PageSection instances remain without hasBodyWrapper, meaning they will render with an extra wrapper div in PF6, potentially breaking CSS targeting and layout.

6.4 CSS Token Migration (pf-v5→pf-v6) semver/goose

What changed in PF6

PF6 introduced a new design token system. CSS variable prefixes changed:

  • Component tokens: --pf-v5-c-* → --pf-v6-c-*
  • Global tokens: --pf-v5-global--* → --pf-t--global--* (new "t" theme token format)
  • CSS class names: .pf-v5-* → .pf-v6-*
  • Chart tokens: --pf-v5-chart-* → --pf-v6-chart-*

pf-codemods — ❌ Zero CSS migration

482 pf-v5 references remain across 2 files:

  • tree-view.css: 480 refs (vendored PF5 CSS)
  • themes.ts: 2 refs (chart theme variables)

These will reference non-existent CSS variables in PF6, causing visual breakage for the tree-view component and chart styling.

semver/goose — ✅ Complete CSS migration

All 482 pf-v5 references eliminated:

/* tree-view.css: 480 variable renames */
--pf-v5-c-tree-view--* → --pf-v6-c-tree-view--*
--pf-v5-global--spacer--sm → --pf-t--global--spacer--sm
--pf-v5-global--Color--100 → --pf-t--global--Color--100
/* ... ~200 unique variable mappings */

/* themes.ts: 2 renames */
--pf-v5-chart-global--FontFamily → --pf-v6-chart-global--FontFamily
--pf-v5-chart-global--letter-spacing → --pf-v6-chart-global--letter-spacing

Impact: This is one of the most impactful migration areas. Without CSS token migration, components using the vendored tree-view CSS will lose all styling in PF6. This is a critical gap in the pf-codemods branch.

6.12 StackItem Migration Bug in semver/goose

What changed in PF6

The StackItem component's component prop specifies the HTML element type to render (e.g., "div", "span"). It does not accept content/children.

pf-codemods — No change (uses main's pattern)

// Main pattern (correct PF6 usage):
<StackItem>{error?.message}</StackItem>
<StackItem><Skeleton fontSize="lg" /></StackItem>

Children are correctly placed as children of StackItem.

semver/goose — ❌ INCORRECT migration

// ErrorAlert.tsx - WRONG:
<StackItem component={error?.message} />
// 'component' expects ElementType ('div','span',etc.)
// NOT a string value like "Something went wrong"

// ListSkeleton.tsx - WRONG:
<StackItem component={<Skeleton fontSize="lg" />} />
// 'component' expects ElementType, NOT JSX element

🔴 Will cause runtime errors. React will attempt to render the error message string as an HTML tag name (e.g., <Something went wrong>) or crash when receiving JSX where a string is expected.

Verdict: The main branch's StackItem usage was already correct for PF6. semver/goose introduced bugs where none existed. semver/goose penalized
