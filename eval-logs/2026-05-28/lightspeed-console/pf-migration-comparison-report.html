⚛️ React Version Upgrade (17 → 18)

Why this matters

The @patternfly/chatbot@6.6.0 package declares "react": "^18 || ^19" as a peer dependency. Running it with React 17 violates the peer dependency contract and may cause runtime issues. The main branch suppresses this with @ts-expect-error: TS2786 comments on every chatbot component usage (12 instances across GeneralPage.tsx and Prompt.tsx).

semver/goose changes

  • Upgraded react and react-dom from 17.0.2 → ^18.3.1
  • Upgraded @types/react from ^17.0.91 → ^18.3.0
  • Upgraded react-redux from 7.2.9 → ^9.3.0 (react-redux 9 supports React 18)
  • Removed overrides.react and overrides.react-dom that forced React 17
  • Removed all 12 @ts-expect-error: TS2786 suppressions (now unnecessary since chatbot types are compatible)

pf-codemods changes

None. React remains at 17.0.2 with overrides.

Side effect analysis

The React 18 upgrade causes eslint-plugin-react-hooks@7.1.1 (resolved via semver) to enforce stricter rules than the 7.0.1 resolved with React 17 types. This surfaces 5 pre-existing issues:

  • 3× react-hooks/set-state-in-effect — changePod() and setKind() called synchronously in effects
  • 2× react-hooks/use-memo — throttle() and debounce() not inline in useCallback

These are pre-existing code patterns; the migration correctly exposed them rather than causing them.

Winner: semver/goose

📐 Modal / ModalBody Changes

What changed in PF6

PF6 promotes the composable Modal from ModalNext. The new Modal requires content to be wrapped in ModalBody, ModalHeader, and ModalFooter subcomponents. The old single-component Modal (with title, actions props) is deprecated to @patternfly/react-core/deprecated.

This app uses a custom Modal wrapper (via react-modal) for most modals. These custom modals aren't directly affected by the PF6 Modal change, but ModalBody should still be used inside them for proper PF6 spacing and semantics.

semver/goose changes

🏷️ Label icon Prop

What changed in PF6

In PF6, the Label component's icon prop is used to render an icon before the label text. Previously, icons could be placed as children alongside text. The PF6 pattern is to use the dedicated icon prop.

Before (main) — AttachmentLabel.tsx

<Label className="ols-plugin__context-label" onClick={onClick} onClose={onClose}> <ResourceIcon kind={kind} /> <span className="ols-plugin__context-label-text">{name}</span>

After (semver/goose)

<Label className="ols-plugin__context-label" onClick={onClick} onClose={onClose} icon={<ResourceIcon kind={kind} />}> <span className="ols-plugin__context-label-text">{name}</span>

Winner: semver/goose — correctly moves icon from children to icon prop.

📋 Dropdown onSelect Signature

What changed in PF6

The composable Dropdown's onSelect callback in PF6 has the signature (event?, value?: DropdownItemProps['value']) => void where value is string | number | undefined. Previously, the second parameter was typed as string.

Before (main) — AttachLogModal.tsx

const onSelect = React.useCallback( (_e: React.MouseEvent<Element, MouseEvent> | undefined, newValue: string) => { closeDropdown(); setContainer(newValue); }, ...);

After (semver/goose)

const onSelect = React.useCallback( (_e: React.MouseEvent<Element, MouseEvent> | undefined, newValue: DropdownItemProps['value']) => { closeDropdown(); setContainer(newValue as string); }, ...);

Applied to both ContainerDropdown and PodDropdown onSelect handlers. The DropdownItemProps import was also added.

Winner: semver/goose

💬 Chatbot @ts-expect-error Cleanup

Background

The main branch has 12 @ts-expect-error: TS2786 suppressions in GeneralPage.tsx (11) and Prompt.tsx (1). These suppress TypeScript errors caused by @patternfly/chatbot components being typed for React 18 while the project uses React 17 types.

Root cause

TS2786 occurs when a component's JSX type is incompatible with the JSX factory. React 17's @types/react uses a different JSX element type than React 18, causing chatbot components to fail type-checking.

semver/goose fix

By upgrading to @types/react@18, the chatbot components' types are now compatible. All 12 @ts-expect-error suppressions were removed cleanly.

🌐 Trans Component i18n Fix

What changed

In ResponseToolModal.tsx, the <Trans> component usage was updated from embedded interpolation {{ name }} to explicit values prop with {'{{name}}'} string template markers.

Before (main)

<Trans> The tool <span className="ols-plugin__code-inline">{{ name }}</span> was requested with arguments <span className="ols-plugin__code-inline">{{ argsFormatted }}</span> but was rejected. </Trans>

After (semver/goose)

<Trans values={{ name, argsFormatted }}> The tool <span className="ols-plugin__code-inline">{'{{name}}'}</span> was requested with arguments <span className="ols-plugin__code-inline">{'{{argsFormatted}}'}</span> but was rejected. </Trans>

Analysis

This change is related to the React 18 upgrade. The old pattern of using {{ name }} directly as JSX children inside <Trans> relied on implicit object spreading behavior that's stricter in React 18's type system. The new pattern explicitly passes values via the values prop, which is the officially recommended react-i18next approach. This is a correctness improvement.

Winner: semver/goose

📊 ImmutableMap .toArray() Fix

What changed

In OlsToolUIs.tsx, .valueSeq() was followed by .toArray() to convert the Immutable.js sequence to a plain array.

Before (main)

.map(({ tool, toolUIComponent }, toolID) => ( <OlsToolUI key={...} tool={tool} toolUIComponent={toolUIComponent} /> )) .valueSeq()}

After (semver/goose)

.map(({ tool, toolUIComponent }, toolID) => ( <OlsToolUI key={...} tool={tool} toolUIComponent={toolUIComponent} /> )) .valueSeq() .toArray()}

Analysis

React 18 is stricter about the types of children it accepts. Immutable.js's Seq type is not a valid React child array in React 18 types. Adding .toArray() converts it to a plain Array, which React accepts. This is a necessary fix for the React 18 upgrade.

Winner: semver/goose
