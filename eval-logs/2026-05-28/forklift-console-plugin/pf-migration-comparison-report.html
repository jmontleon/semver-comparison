­¤Ü© CRITICAL: StackItem/SplitItem component= Prop Misuse (Branch 2)

The Problem

Branch 2 systematically moves children content (text strings, JSX elements) from being children of <StackItem> and <SplitItem> into their component= prop. This is semantically wrong.

The component prop on PatternFly layout components specifies the HTML element type to render (e.g., "div", "span", "li"). It does NOT accept content/children.

Examples of the bug (~88 instances)

// BEFORE (main ŌĆö correct)
<StackItem className="text-muted">
  {t('Rules with "Preferred" condition will stack with an "AND" relation between them.')}
</StackItem>

// AFTER (Branch 2 ŌĆö BROKEN)
<StackItem className="text-muted"
  component={t('Rules with "Preferred" condition will stack with an "AND" relation between them.')} />
// BEFORE (main ŌĆö correct)
<StackItem>
  {t('Defines the CPU limits allocated to the main container...')}
</StackItem>

// AFTER (Branch 2 ŌĆö BROKEN)
<StackItem component={t('Defines the CPU limits allocated to the main container...')} />

Impact

  • UI Text Disappears ŌĆö The text content that was previously rendered as children will no longer appear. Instead, React will try to use the text string as an HTML element type, which will either render an empty unknown element or throw a runtime error.
  • ~88 affected locations across help content components, description texts, project select fields, settings help content, storage product fields, and more.
  • This is the single biggest reason Branch 2 cannot be merged.

Root Cause

This appears to be an AI agent's misunderstanding of what the component prop does. The agent likely saw PatternFly migration guidance about "moving content to props" (which applies to things like EmptyState's titleText prop or Button's icon prop) and over-applied the pattern to layout components where it doesn't apply.

Winner: Branch 1 ŌĆö does not have this issue.
